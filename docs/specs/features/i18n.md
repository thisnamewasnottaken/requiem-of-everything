# Feature Spec: Internationalisation (i18n)

## Overview

Requiem of Everything supports multiple languages for all UI chrome (labels, buttons, headings, help text, tooltips, accessibility labels). Data content (composer biographies, composition descriptions, event descriptions) remains in the original language, with the i18n infrastructure supporting future content localisation.

## Supported Locales

| Locale | Language | Status |
|--------|----------|--------|
| `en-GB` | English (United Kingdom) | Default, fallback |
| `fr-FR` | French (France) | First target |
| `af-ZA` | Afrikaans (South Africa) | Second target |

## Architecture

- **Library**: `react-i18next` (wrapping `i18next`)
- **Backend**: `i18next-http-backend` ‚Äî loads translation JSON from `public/locales/{lng}/translation.json`
- **Detection**: `i18next-browser-languagedetector` ‚Äî detects browser language, persists user choice in `localStorage`
- **Fallback**: `en-GB` is always the fallback language
- **Namespace**: Single `translation` namespace (default)


## Language Switcher

A `<select>` dropdown in the app header (right side, before the Filters button) allowing the user to switch language. Persisted via `localStorage` key `i18nextLng`.

## Translation Scope

### Translated (UI chrome)
- All component labels, headings, button text
- Tooltips and `aria-label` attributes
- Help panel content (all sections)
- Filter panel labels and placeholders
- Era names, genre names, event category names (as display values)
- Error/empty state messages

### Content Localisation (data content ‚Äî via namespaces)

Content data is localised through dedicated i18next namespaces. The base data files (`src/data/*.json`) remain canonical and contain non-translatable fields (IDs, years, URLs) alongside English defaults. Translatable fields are mirrored into namespace JSON files under `public/locales/{lng}/`.

#### Namespaces

| Namespace | Source data file | Localised fields |
|-----------|-----------------|------------------|
| `translation` | ‚Äî | UI chrome strings (existing) |
| `composers` | `src/data/composers.json` | `name`, `shortName`, `biography`, `birthPlace`, `nationality` |
| `compositions` | `src/data/compositions.json` | `title`, `description`, `significance`, `instrumentation` |
| `events` | `src/data/events.json` | `title`, `description`, `musicalSignificance`, `region` |
| `eras` | `src/data/eras.json` | `name`, `description`, `characteristics` |

```json

    "name": "Johann Sebastian Bach",

    "nationality": "German"
  }

```

#### Fallback behaviour

- Missing namespace file ‚Üí i18next falls back to `en-GB` (via `fallbackLng`)
- Missing key within a file ‚Üí `defaultValue` parameter in `t()` returns the English base data value
- i18next `ready === false` ‚Üí hooks return raw English base data (no merge)

#### Hook integration

Data hooks in `src/hooks/useData.ts` use `useTranslation(namespace)` to obtain a namespace-scoped `t` function. Each hook maps over base data, replacing translatable fields with `t('entityId.field', { defaultValue: baseValue })`. Results are memoised with `useMemo` to avoid recomputation unless the language changes.

Internal helper hooks (`useTranslatedComposers`, `useTranslatedCompositions`, etc.) compute the full translated arrays. Public hooks delegate to these helpers, applying any filtering or lookup on top.

#### i18n configuration

```typescript
ns: ['translation', 'composers', 'compositions', 'events', 'eras'],
defaultNS: 'translation',
```

The `i18next-http-backend` loadPath `locales/{{lng}}/{{ns}}.json` automatically resolves each namespace per locale.

## Accessibility

- Language switcher has `aria-label="Select language"`
- `<html lang>` attribute updates when language changes
- Screen readers announce language change via `aria-live` region
