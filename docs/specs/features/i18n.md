# Feature Spec: Internationalisation (i18n)

## Overview

Requiem of Everything supports multiple languages for all UI chrome (labels, buttons, headings, help text, tooltips, accessibility labels). Data content (composer biographies, composition descriptions, event descriptions) remains in the original language, with the i18n infrastructure supporting future content localisation.

## Supported Locales

| Locale  | Language                 | Status            |
| ------- | ------------------------ | ----------------- |
| `en-GB` | English (United Kingdom) | Default, fallback |
| `fr-FR` | French (France)          | First target      |
| `af-ZA` | Afrikaans (South Africa) | Second target     |

## Architecture

- **Library**: `react-i18next` (wrapping `i18next`)
- **Backend**: `i18next-http-backend` — loads translation JSON from `public/locales/{lng}/translation.json`
- **Detection**: `i18next-browser-languagedetector` — detects browser language, persists user choice in `localStorage`
- **Fallback**: `en-GB` is always the fallback language
- **Namespaces**: 7 namespaces — `translation` (default), `composers`, `compositions`, `events`, `eras`, `terms`, `instruments`

## Translation File Structure

```
public/locales/
├── en-GB/translation.json
├── fr-FR/translation.json
└── af-ZA/translation.json
```

## Translation Key Convention

Hierarchical dot-notation keys organised by component/section:

```json
{
  "app": { "title": "...", "subtitle": "..." },
  "filters": { "title": "...", "search": "...", ... },
  "help": { "title": "...", "shortcuts": { ... }, ... },
  "composer": { "focusTimeline": "...", ... },
  "composition": { "instrumentation": "...", ... },
  "comparison": { "comparing": "...", ... },
  "common": { "close": "...", "clear": "...", ... },
  "eras": { "renaissance": "...", ... },
  "genres": { "symphony": "...", ... },
  "categories": { "war": "...", ... }
}
```

## Language Switcher

A `<select>` dropdown in the app header (right side, before the Filters button) allowing the user to switch language. Persisted via `localStorage` key `i18nextLng`.

## Translation Scope

### Translated (UI chrome)

- All component labels, headings, button text
- Tooltips and `aria-label` attributes
- Help panel content (all sections)
- Filter panel labels and placeholders
- Era names, genre names, event category names (as display values)
- Error/empty state messages

### Content Localisation (data content — via namespaces)

Content data is localised through dedicated i18next namespaces. The base data files (`src/data/*.json`) remain canonical and contain non-translatable fields (IDs, years, URLs) alongside English defaults. Translatable fields are mirrored into namespace JSON files under `public/locales/{lng}/`.

#### Namespaces

| Namespace      | Source data file             | Localised fields                                              |
| -------------- | ---------------------------- | ------------------------------------------------------------- |
| `translation`  | —                            | UI chrome strings (existing)                                  |
| `composers`    | `src/data/composers.json`    | `name`, `shortName`, `biography`, `birthPlace`, `nationality` |
| `compositions` | `src/data/compositions.json` | `title`, `description`, `significance`, `instrumentation`     |
| `events`       | `src/data/events.json`       | `title`, `description`, `musicalSignificance`, `region`       |
| `eras`         | `src/data/eras.json`         | `name`, `description`, `characteristics`                      |
| `terms`        | `src/data/terms.json`        | Term names, descriptions, and related content                 |
| `instruments`  | `src/data/instruments.json`  | Instrument names, descriptions, and related content           |

#### Namespace file structure

```
public/locales/
├── en-GB/
│   ├── translation.json      # UI strings
│   ├── composers.json         # keyed by composer id
│   ├── compositions.json      # keyed by composition id
│   ├── events.json            # keyed by event id
│   ├── eras.json              # keyed by era id
│   ├── terms.json             # keyed by term id
│   └── instruments.json       # keyed by instrument id
├── fr-FR/
│   ├── translation.json
│   ├── composers.json
│   └── ...
└── af-ZA/
    └── ...
```

Each content namespace file is a flat object keyed by entity ID:

```json
{
  "johann-sebastian-bach": {
    "name": "Johann Sebastian Bach",
    "shortName": "Bach",
    "biography": "...",
    "birthPlace": "Eisenach, Saxe-Eisenach",
    "nationality": "German"
  }
}
```

#### Fallback behaviour

- Missing namespace file → i18next falls back to `en-GB` (via `fallbackLng`)
- Missing key within a file → `defaultValue` parameter in `t()` returns the English base data value
- i18next `ready === false` → hooks return raw English base data (no merge)

#### Hook integration

Data hooks in `src/hooks/useData.ts` use `useTranslation(namespace)` to obtain a namespace-scoped `t` function. Each hook maps over base data, replacing translatable fields with `t('entityId.field', { defaultValue: baseValue })`. Results are memoised with `useMemo` to avoid recomputation unless the language changes.

Internal helper hooks (`useTranslatedComposers`, `useTranslatedCompositions`, etc.) compute the full translated arrays. Public hooks delegate to these helpers, applying any filtering or lookup on top.

#### i18n configuration

```typescript
ns: ['translation', 'composers', 'compositions', 'events', 'eras', 'terms', 'instruments'],
defaultNS: 'translation',
```

The `i18next-http-backend` loadPath `locales/{{lng}}/{{ns}}.json` automatically resolves each namespace per locale.

## Accessibility

- Language switcher has `aria-label` (translation key `app.languageSelect`)
- `<html lang>` attribute updates when language changes
- Screen readers announce language change via `aria-live` region

## Translation Completeness Enforcement

- A Git pre-commit hook enforces strict i18n checks plus test coverage before a commit is accepted.
- Strict i18n command: `npm run i18n:strict`
- Strict mode fails when placeholder values prefixed with `__MISSING_TRANSLATION__:` are present in locale files.
- Local fast-path test strategy: run `vitest related --run --passWithNoTests` for staged source/config files.
- CI/full mode test strategy: run `npm run test:run`.
- Force full local tests by setting `PRECOMMIT_TEST_SCOPE=full`.
- Installation command: `npm run hooks:install`
- Commit policy: commits are blocked when missing keys are detected, placeholder translation values remain, or tests fail.
- Repair workflow: run `node ./.github/skills/i18n-completeness/scripts/check-and-fill.mjs --apply`, replace placeholder values with real translations, then commit.
